90-Day Entry Plan | Senior Autonomy Integration Engineer

Enter through evidence. Earn leverage through closure.

The plan is designed to learn Latitude's actual architecture, toolchain, integration rhythm, and decision boundaries before prescribing changes. Each phase increases direct contribution while preserving domain ownership.

Days 0-30 | Learn the system
Technical immersionSet up the C++/Python build and test environment; learn repository structure, platform abstractions, configuration management, telemetry, validation, on-vehicle compute constraints, and safety interfaces.
Integration flow mapTrace one feature or platform change from configuration through compute/network, sensors, autonomy software, test evidence, release posture, and ownership handoff.
Failure evidence mapReview representative bring-up, regression, intermittent, and configuration issues. Identify how symptoms become hypotheses and where ambiguity persists.
Relationship mapPair with autonomy, systems, safety, test, platform, compute, networking, sensor, and configuration owners.
First bounded contributionOwn a contained integration issue or tooling improvement with a senior partner; document assumptions, evidence, experiments, repair, and durable closure mechanism.
Days 31-60 | Improve the signal
Own bounded integration workDrive selected issues from reproduction through closure, keeping the next experiment, evidence gap, owner, and release posture explicit.
Strengthen observabilityImprove synchronized logs, trace correlation, platform/configuration identifiers, or diagnostics where missing evidence forces guesswork.
Make configuration comparableSupport machine-readable comparison of software, firmware, calibration, network, environment, and platform identities against known-good references.
Improve blocker communicationUse concise integration readouts that state symptom, invariant, evidence, hypotheses, next experiment, owner, and release impact.
Convert closure into protectionAdd targeted regression checks, compatibility controls, validation evidence, or interface documentation for repeated failure patterns.
Days 61-90 | Strengthen the mechanism
Pattern reviewCluster recurring integration failures by violated invariant, missing evidence, configuration drift, ownership boundary, and escaped-regression mechanism.
Reusable closure mechanismsPropose high-leverage checks, templates, dashboards, or tooling changes grounded in observed work.
Integration health viewMake blocker age, reproducibility, configuration traceability, release evidence, escaped regression, and recurrence visible without vanity metrics.
Platform readiness supportApply the model to one new-platform or major-feature integration path and document what should become easier for the next rollout.
Operating principles
Evidence over ego
Pair before prescribing
Respect domain ownership
Automate repeatable checks
Make release posture explicit
Leave the system easier to explain
